# ADR-0002: Unify Playwright Visual Baselines on Linux via Docker

**Date:** 2026-05-23
**Status:** Proposed

## Context

`apps/ui/` ships Playwright visual regression tests for Storybook stories. The repo currently maintains baselines for two platforms:

- 30 `*-chromium-darwin.png` files (regenerated on contributor Macs).
- 30 `*-chromium-linux.png` files (regenerated by CI).

That is **60 PNGs across 30 logical assertions** — every visual change touches two files per story.

We have hit the bootstrap friction twice. On #29 we introduced the Linux baselines from scratch via a CI bootstrap commit. PR #40 (the visual refresh) was forced into the same two-step: regenerate Darwin locally, delete Linux, push, wait for CI to capture the new Linux PNGs, push a second "Bootstrap Linux baselines for #..." commit. This pattern shows up across history (#29, #34/#37 refresh, #40). Each round is a contributor-blocking CI round-trip on a documentation-only commit.

There is also a slow-moving divergence risk: Linux font hinting (FreeType + the Playwright container's font stack) drifts from Darwin's CoreText rendering as upstream packages move. The two baselines depict the same story but with subtly different glyph anti-aliasing, and the gap can in principle widen past the configured `maxDiffPixelRatio` (1–3%) without anyone changing a line of UI code. The current single-Chromium project (`playwright.config.ts`) keeps the matrix small, but the per-OS asymmetry is structural, not a config knob.

CLAUDE.md is explicit that this is a teaching-grade, portfolio-worthy repo optimised for clarity, testability, and demonstrability over raw performance. Visual-test infrastructure that costs two commits per visual change and silently encodes platform drift is a tax on exactly those properties.

Issue #41 proposes `npm run baseline:linux` — a one-command Docker wrapper that regenerates Linux baselines from a Mac. That tooling is the missing piece that makes a Linux-only world ergonomic from a non-Linux host.

## Options considered

### (a) Status quo — keep both per-platform baselines

Keep regenerating Darwin baselines locally on Macs and Linux baselines on CI.

- **Pro:** No migration. Fast inner-loop on Mac because `npx playwright test --update-snapshots` writes the file the OS already cares about.
- **Pro:** Local `npm run test:visual` on Mac stays click-instant — no Docker dependency.
- **Con:** 60 PNGs in git for 30 logical assertions; every visual PR touches both halves or relies on the CI two-step.
- **Con:** Bootstrap friction is permanent (#29 → #40 → next visual refresh → ...).
- **Con:** Silent platform drift risk on font hinting; the Mac baseline can pass forever while Linux quietly creeps toward the threshold.
- **Con:** "Which baseline is canonical" is undefined. CI is the gatekeeper, but contributors design against the Darwin file.

### (b) Linux-only, via the Playwright Docker image

Drop the `*-chromium-darwin.png` files. All baselines live and breathe as Linux PNGs. Contributors regenerate them through `npm run baseline:linux` (issue #41), which pins the same Playwright Docker tag as `package.json`.

- **Pro:** One canonical platform. CI and local agree pixel-for-pixel because they share an image hash.
- **Pro:** Bootstrap two-step disappears. A visual PR contains the new Linux PNGs in the same commit that changes the UI; CI re-runs the assertion, doesn't capture.
- **Pro:** Halves the baseline surface area (60 → 30 files).
- **Pro:** Eliminates the font-hinting drift risk by construction — there is no second platform to drift from.
- **Con:** Local visual iteration on Mac now requires Docker. First run pulls the Playwright image (~1 GB).
- **Con:** Mac contributors can no longer eyeball failures with a familiar local `npx playwright test --update-snapshots`; the command becomes "spin up the container."
- **Con:** Hard dependency on #41 landing before the Darwin files can be deleted.

### (c) Hybrid — Darwin for inner-loop, Linux as canonical for merge gating

Keep both, but officially designate Linux as the authoritative baseline. Darwin baselines are advisory; CI gates on Linux only.

- **Pro:** Preserves the Mac inner-loop ergonomics of (a).
- **Pro:** Resolves the "which platform wins" ambiguity.
- **Con:** Does **not** reduce the file count or maintenance cost — still 60 PNGs.
- **Con:** Does **not** eliminate the bootstrap two-step; Linux baselines still come from CI unless #41 is also adopted, at which point we are doing (b)'s work to keep (a)'s files.
- **Con:** Introduces a new failure mode: Darwin baseline passes locally, Linux baseline fails on CI, contributor has to context-switch into the canonical platform anyway. We've just spent disk and review-attention on a second baseline that doesn't gate.

## Decision

**Adopt option (b): Linux is the single canonical platform for Playwright visual baselines.**

Rationale, anchored in CLAUDE.md:

- CLAUDE.md frames this project as "optimized for clarity, testability, and demonstrability." Duplicate baselines for two operating systems are anti-clarity — every PR reviewer has to mentally diff twice — and a hidden source of test-suite non-determinism.
- "Every protocol round trip is logged at a level that makes the trace inspectable for demos" is the project's testing ethos. The CI loop is a kind of round trip; the current two-step bootstrap muddies that trace with a second commit whose purpose is "capture, don't verify." Linux-only collapses that to one commit per visual change.
- The Mac-local Docker cost is paid once per visual PR and recouped immediately by not waiting on a CI bootstrap round-trip. For a teaching repo, the lower bound on iteration latency is what matters; we are not optimising for hot-edit fluidity.
- Option (c) only makes sense if local Mac iteration is on the critical path for many contributors. It isn't — this is a single-maintainer portfolio repo with an AFK agent fleet that all runs in Linux.

## Consequences

**Contributor workflow (after this ADR + #41):**

- Visual change touches `apps/ui/`. Contributor runs `npm run baseline:linux` from `apps/ui/`, which spins up the pinned Playwright Docker image and writes `*-chromium-linux.png` files.
- Contributor commits the new PNGs in the same commit as the UI change. No "Bootstrap Linux baselines for #..." follow-up commit.
- CI runs `npm run test:visual` against the committed baselines and either passes or fails — never captures.
- Day-to-day `npm run test:visual` on Mac (without `--update-snapshots`) continues to work for the "did I break a story?" check; only the regen step gates on Docker.

**Migration plan:**

- This ADR is documentation only. It changes no code, no scripts, no PNGs.
- Once #41 lands (`npm run baseline:linux` script + README update), a follow-up PR will:
  1. Delete all 30 `*-chromium-darwin.png` files under `apps/ui/src/tests/snapshots/visual.spec.ts-snapshots/`.
  2. Confirm that `npm run baseline:linux` followed by `npm run test:visual` passes locally.
  3. Reference this ADR in the commit message.
- Until that follow-up PR lands, the status-quo two-step remains in effect; the ADR does not retroactively block anything.

**What we accept in trade:**

- Mac contributors must have Docker available before regenerating baselines. This is a real cost for a casual outside contributor but a negligible one for the current team.
- First-time Docker image pull is slow. Mitigated by pinning the tag so it caches.
- We are betting that Linux font-hinting churn inside the pinned Playwright image is slower than the cross-platform drift we'd otherwise face. This is a defensible bet: the image is reproducible, the pin is explicit.

## Alternatives considered

See "Options considered" above. (a) and (c) are documented for the record and explicitly rejected.
